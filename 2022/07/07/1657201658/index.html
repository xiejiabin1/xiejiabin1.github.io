高并发场景乐观锁与悲观锁

高并发 Java锁
Java锁
发布日期:   2022-07-07
更新日期:   2022-07-07
文章字数:   2.9k
阅读时长:   10 分
阅读次数:  

当程序中可能出现并发的情况时，就需要保证在并发情况下数据的准确性，以此确保当前用户和其他用户一起操作时，所得到的结果和他单独操作时的结果是一样的。这就叫做并发控制。并发控制的目的是保证一个用户的工作不会对另一个用户的工作产生不合理的影响。

没有做好并发控制，就可能导致脏读、幻读和不可重复读等问题。实现并发控制的主要手段分为乐观并发控制和悲观并发控制两种。无论是悲观锁还是乐观锁，都是人们定义出来的概念，可以认为是一种思想。

一、悲观锁

1、理解

  在高并发场景下，当要对数据库中的一条数据进行修改的时候，为了避免同时被其他人修改，最好的办法就是直接对该数据进行加锁以防止并发。

  为什么叫做悲观锁呢？因为这是一种对数据的修改抱有悲观态度的并发控制方式。我们一般认为数据被并发修改的概率比较大，所以每次在拿数据的时候都会上锁。

  • 以数据库锁机制实现悲观锁：悲观锁的实现，往往依靠数据库提供的锁机制，也只有数据库层提供的锁机制才能真正保证数据访问的排他性，否则，即使在本系统中实现了加锁机制，也无法保证外部系统不会修改数据。
  • 使用 synchronized 关键字实现：synchronized是有序性的，一个变量在同一时刻只允许一条线程对其进行 lock 操作。

2、以数据库锁机制实现悲观锁

场景：有用户A和用户B，在同一家店铺去购买同一个商品，但是商品的可购买数量只有一个。在不加锁的情况下，如果用户A和用户B同时下单，就会报错。

  传统的关系型数据库使用这种锁机制，比如行锁、表锁、读锁、写锁等，都是在操作之前先上锁（以下内容以MySQL数据库为例）。

2.1、行锁

  行锁：操作时只锁某一行，不对其他行有影响。行锁偏向InnoDB存储引擎，开销大，加锁慢，会出现死锁，锁定粒度最小，发生锁冲突的概率最低，并发度也很高。InnoDB与MyISAM的最大不同有两点：一是支持事务，二是采用了行级锁。由于MySQL的行锁是针对索引加的锁，不是针对记录加的锁，所以虽然是访问不同行的记录，但是如果是使用相同的索引键，是会出现锁冲突的。

  创建数据库：并且设置数据库引擎为：InnoDB

CREATE TABLE `test_table` (
  `id` INT(11) NOT NULL AUTO_INCREMENT,
  `name` VARCHAR(40) DEFAULT NULL,
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8
  • 1：需要关闭mysql自动提交
SET autocommit = 0;
  • 2：session1为某一行数据添加锁（FOR UPDATE）
SELECT * FROM test_table WHERE id = 1 FOR UPDATE;
  • 3：session2为另一行数据添加锁。虽然session2访问的是和session1不同的记录，但是因为使用了相同的索引，所以需要等待锁：
SELECT * FROM test_table WHERE id = 2 FOR UPDATE;

2.2、读锁（表锁）

  读锁（共享锁）：针对同一份数据，多个操作可以同时进行而不会相互影响。

  创建数据库：并且设置数据库引擎为：MyISAM

CREATE TABLE `test_table` (
  `id` INT(11) NOT NULL AUTO_INCREMENT,
  `name` VARCHAR(40) DEFAULT NULL,
  PRIMARY KEY (`id`)
) ENGINE=MYISAM DEFAULT CHARSET=utf8
  • 给表test_table添加读锁
LOCK TABLE `test_table` READ
  • 当有两个session用户访问同一个表test_table时，session1给test_table添加读锁时发现：
  • session1用户可以查询这张表的数据，session2也可以查询这张表的数据
  • session1用户修改这张表失败了，因为现在把这张表锁了，所以修改操作会出错

  • session1查看别的锁时，发现也失败了，因为当锁住了test_table这张表时，在对这张表操作完之后应该释放掉这把锁才能操作别的表

  • session2想要修改这张已经被锁住的test_table表的内容时，会陷入阻塞中，因为test_table的锁还有释放不能进行修改操作
  • unlock tables 解除读锁

2.3、写锁（表锁）

  写锁（排他锁）：当前写操作没有完成前，他会阻断其他写锁和读锁

  • 1：首先进行加锁：
LOCK TABLE `test_table` WRITE
  • 2：进行查找表test_table里的内容，发现可以查出来
  • 3：当想要select别的表和修改mylock表时，就会报错，理由和读锁是一样，要先释放才可以查找别的表
  • 4：当session2想要查看被写锁锁住的表时，会陷入阻塞，只有当释放锁时才可以select出来

3、使用 synchronized 关键字实现

  synchronized有一下三种使用方式

  • 修饰实例方法，为当前实例加锁，进入同步方法前要获得当前实例的锁。
  • 修饰静态方法，为当前类对象加锁，进入同步方法前要获得当前类对象的锁。
  • 修饰代码块，指定加锁对象，对给定对象加锁，进入同步代码块前要获得给定对象的锁。

  有一个要注意的地方是对静态方法的修饰可以和实例方法的修饰同时使用，不会阻塞，因为一个是修饰的Class类，一个是修饰的实例对象。

二、乐观锁

1、理解

乐观锁是相对悲观锁而言的，乐观锁假设数据一般情况不会造成冲突，操作数据时不会对操作的数据进行加锁，只有到数据提交的时候才通过一种机制来验证数据是否存在冲突。

乐观锁实现方式乐观锁不需要借助数据库的锁机制，主要就是两个步骤：冲突检测和数据更新。

  • CAS 实现：Java 中java.util.concurrent.atomic包下面的原子变量使用了乐观锁的一种 CAS 实现方式。
  • 版本号控制：一般是在数据表中加上一个数据版本号 version 字段或者时间戳 timestamp，表示数据被修改的次数。当数据被修改时，version 值会 +1。当线程 A 要更新数据时，在读取数据的同时也会读取 version 值，在提交更新时，若刚才读取到的 version 值与当前数据库中的 version 值相等时才更新，否则重试更新操作，直到更新成功。

2、CAS实现

  CAS意思是：比较并替换。CAS需要有3个操作数：内存地址V，旧的预期值A，即将要更新的目标值B。

  CAS指令执行时，当且仅当内存地址V的值与预期值A相等时，将内存地址V的值修改为B，否则就什么都不做。整个比较并替换的操作是一个原子操作。下面以AutoInteger为例：

@Test
public void test20() throws InterruptedException {
    for (int i = 1; i <= 5; i++) {
        MyThread thrend = new MyThread("线程" + i);
        Thread thread = new Thread(thrend);
        thread.start();
    }
    Thread.sleep(2000);
    System.out.println(MyCount.count);
}


static class MyThread implements Runnable {
    private String name;
    MyThread(String threadName) {
        this.name = threadName;
    }

    @Override
    public void run() {
        for (int i=0;i<20;i++) {
            MyCount.count++;
            System.out.println(this.name + "count："+ MyCount.count);
        }
    }
}

private static class MyCount {
    static int count = 0;
}

  以上内容为：一个共享变量count=1，开5个线程，每个线程加20次，结果一般来说都会小于100（数量大了结果一般都不一致）。原因是在出现多次同一时间多个线程拿到了同一个count，造成了数量和数值错误。

  现在修改一个代码，将int变成AtomicInteger，如下：

@Test
public void test20() throws InterruptedException {
    for (int i = 1; i <= 5; i++) {
        MyThread thrend = new MyThread("线程" + i);
        Thread thread = new Thread(thrend);
        thread.start();
    }
    Thread.sleep(2000);
    System.out.println(MyCount.count);
}


static class MyThread implements Runnable {
    private String name;
    MyThread(String threadName) {
        this.name = threadName;
    }

    @Override
    public void run() {
        for (int i=0;i<20;i++) {
            // 加1方法
            MyCount.count.getAndIncrement();
            System.out.println(this.name + "count："+ MyCount.count);
        }
    }
}

private static class MyCount {
    static AtomicInteger count = new AtomicInteger(0);
}

结果显示：每次结果都是100，AtomicInteger加1方法的源码如下：

public final int getAndIncrement() {
    // 第一个参数当前对象地址，第二个参数数据偏移量，第三个参数每次指定默认加1
    return unsafe.getAndAddInt(this, valueOffset, 1); 
}
// 这个方法使用的就是CAS,核心在于循环比较内存里面的值和当前值是否相等，如果相等就用新值覆盖
public final int getAndAddInt(Object var1, long var2, int var4) {  
    int var5;   
    do {
        //如果a，b线程同时执行这个方法，a线程拿到值1后cpu执行时间到了挂起，b开始执行，也拿到1，但是没有挂起，接着将值变成了2
        var5 = this.getIntVolatile(var1, var2);  
    //这个时候a线程恢复执行，去比较的时候发现手上的1 和内存里面的值2不等，这个时候他要进行下一个循环，占用cpu
    } while(!this.compareAndSwapInt(var1, var2, var5, var5 + var4));  
    return var5;
}

除了AtomicInteger，还有AtomicLong、AtomicBoolean…..都在java.util.current.atomic包下面，采用了CAS机制来实现加锁。但是CAS容易引发ABA问题：

  • 比如说线程一从数据库中取出库存数 3，这时候线程二也从数据库中取出库存数 3，并且线程二进行了一些操作变成了 2。
  • 然后线程二又将库存数变成 3，这时候线程一进行 CAS 操作发现数据库中仍然是 3，然后线程一操作成功。
  • 尽管线程一的 CAS 操作成功，但是不代表这个过程就是没有问题的。

CAS机制类似于以下SQL：

# 查询出商品库存 quantity=3
select quantity from goods where id = 1;
# 修改商品库存为2
update goods set quantity=quantity-1 where id=1 and quantity=3

3、版本控制实现

用版本控制实现乐观锁可以解决CAS可能造成的ABA问题。乐观锁每次在执行数据修改操作时，都会带上一个版本号，一旦版本号和数据的版本号一致就可以执行修改操作并对版本号执行 +1 操作，否则就执行失败。因为每次操作的版本号都会随之增加，所以不会出现 ABA 问题。除了 version 以外，还可以使用时间戳，因为时间戳天然具有顺序递增性。

比如以下SQL：

# 查询出商品库存和该数据的版本号 quantity=3 version=1
select quantity,version from goods where id = 1;
# 修改商品库存为2
update goods set quantity=quantity-1 ,version=version+1 where id=1 and version=1

以上 SQL 其实还是有一定的问题的，就是一旦遇上高并发的时候，就只有一个线程可以修改成功，那么就会存在大量的失败，总让用户感知到失败显然是不合理的。所以，还是要想办法减少乐观锁的粒度。一个比较好的建议，就是减小乐观锁力度，最大程度的提升吞吐率，提高并发能力。如下：

# 查询出商品库存和该数据的版本号 quantity=3
select quantity,version from goods where id = 1;
# 修改商品库存为2
update goods set quantity=quantity-1 where id=1 and quantity-1>0
